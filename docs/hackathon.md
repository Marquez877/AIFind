# 1. История. Зачем это важно

> «Тот, кто не знает своего прошлого, не имеет будущего.»
> — кыргызская пословица

В 1937 году учителя Асана Байтемирова арестовали на рассвете. Обвинение — «враг народа». Через три месяца расстрел. Его жене не сообщили ни когда, ни где. Дочь узнала правду через 60 лет — из случайно найденного архивного листа.

Асан Байтемиров — один из десятков тысяч. В период с 1918 по 1953 год тысячи жителей Кыргызстана были расстреляны, сосланы или вычеркнуты из официальной истории. Их имена — в пыльных папках, разрозненных списках, чудом уцелевших документах.

### Задача хакатона

Построить платформу, которая вернёт этим людям имена: цифровой архив с ИИ-ассистентом, который отвечает на вопросы по документам и помогает исследователям, журналистам и потомкам находить правду.

# 2. Задача

## Создать веб-платформу «Голос из архива»

Платформа объединяет два модуля в единый продукт:

1. Архив исторических данных
2. ИИ-ассистент, который работает с документами этого архива

Это не два отдельных сервиса — это одна система.

### 01. Хранить и показывать

Карточки репрессированных:

* Имя
* Годы жизни
* Регион
* Обвинение
* Биография
* Прикреплённые документы

Любой пользователь может добавлять записи и указывать источники.

### 02. Искать и фильтровать

Поиск:

* По имени
* По году
* По региону
* По типу обвинения

Система должна выявлять и предотвращать появление дублирующихся записей.

### 03. Отвечать на вопросы

Встроенный ИИ-ассистент:

* Принимает документы архива
* Находит нужные фрагменты
* Отвечает через языковую модель

Это RAG — Retrieval-Augmented Generation.

---

# 3. Что такое RAG и как это реализовать

RAG (Retrieval-Augmented Generation) — подход, при котором языковая модель отвечает не из общих знаний, а на основе конкретных документов, загруженных пользователем.

## Этапы реализации

### 1. Загрузка

Пользователь загружает документ:

* Биографию
* Архивное дело
* Свидетельство

Система разбивает текст на фрагменты (чанки) по 500–1000 символов.

### 2. Поиск

После получения вопроса система ищет наиболее подходящие фрагменты.

Варианты реализации:

* Базовый: все фрагменты передаются в prompt
* Продвинутый: используются эмбеддинги и top-3 по косинусному сходству

### 3. Ответ

Найденные фрагменты вместе с вопросом отправляются в языковую модель:

* OpenAI
* Anthropic

Модель формирует ответ на основе реальных документов архива.

# 4. Требования к продукту

## 4.1 Обязательный минимум

Без реализации всех пунктов ниже проект не допускается к оценке:

* Карточка человека:

  * имя
  * годы жизни
  * регион
  * обвинение
  * краткое описание
* Добавление записей с базовой проверкой на дубли:

  * совпадение имени
  * совпадение года рождения
* Поиск по имени и фильтр по категории или региону
* Загрузка документов `.txt` / `.md` через POST-эндпоинт
* Передача документов в LLM с вопросом пользователя
* Чат-интерфейс: вопрос → ответ на основе документов
* Публичный GitHub-репозиторий с README

## 4.2 Рекомендуемые улучшения

Эти функции повышают оценку по критериям «Качество ИИ-поиска» и «Дизайн»:

* Семантический поиск по чанкам через векторные эмбеддинги

  * top-3
  * косинусное сходство
* Показ источников под ответом
* Полноценный чат-интерфейс с историей диалога
* Расширенный поиск:

  * по имени
  * региону
  * году репрессии
  * типу обвинения
* Система модерации:

  * «верифицировано»
  * «требует проверки»
* Визуализация данных:

  * карта регионов
  * временная шкала событий
* Drag & Drop загрузка файлов

## 4.3 Бонусные функции

Каждая функция добавляет баллы независимо:

| Функция                  | Описание                                                                |
| ------------------------ | ----------------------------------------------------------------------- |
| Умное обнаружение дублей | Нечёткий поиск, сравнение через эмбеддинги, предложение похожих записей |
| Поддержка PDF            | Работа с PDF-файлами                                                    |
| Docker Compose           | Запуск проекта одной командой                                           |
| JWT-авторизация          | Роли: модератор и пользователь                                          |
| Мультиязычность          | Кыргызский, русский, английский                                         |
| История диалогов         | Сохранение чатов в БД                                                   |
| Unit-тесты               | Минимум 5 тестов                                                        |
| Адаптивный дизайн        | Мобильная версия                                                        |
| SSE                      | Стриминг ответа посимвольно                                             |

# 5. Стек технологий

Команды свободны в выборе технологий.

| Компонент       | Варианты                                                     |
| --------------- | ------------------------------------------------------------ |
| Backend         | Go (Fiber / Chi), Python (FastAPI), Node.js (Express)        |
| Frontend        | Next.js, React, Vue, Svelte                                  |
| База данных     | SQLite, PostgreSQL                                           |
| LLM             | OpenAI API, Anthropic API                                    |
| Векторный поиск | Косинусное сходство вручную или любая open-source библиотека |

> Важно: API-ключи для языковых моделей выдают организаторы. Количество токенов ограничено — не делайте лишних вызовов и кэшируйте эмбеддинги.

# 8. Правила

* Команда: 2–4 человека
* Состав фиксируется до начала хакатона
* Разрешены AI-ассистенты:

  * GitHub Copilot
  * Claude
  * ChatGPT
* Каждый участник должен объяснить свой код
* Плагиат между командами = дисквалификация
* Коммиты должны быть осмысленными
* Разрешены любые open-source библиотеки и фреймворки
* Проект обязан запускаться по README
* Если жюри не сможет запустить проект — это снизит оценку
